#100 - Detect Moves + GPT-5.1 Repair

What & Why
Three confirmed bugs are breaking the Move Calibration flow and AI pipeline:

Bug A — avgMovePct double multiplication (698% display) The calibrationAggregator.ts already converts movePct fractions to percentages (movePcts = moves.map(m => m.movePct * 100)) before computing avgMovePct. The UI then multiplies again: aggregate.overall.avgMovePct * 100. Result: a real 6.98% average shows as 698.4%. Same double-conversion exists in extractionPass.ts for every avgMovePct and medianMovePct it writes into the calibration profile.

Bug B — BOOM300/CRASH300 mostly "unknown" family labels moveLabeler.ts has no symbol awareness. Its generic thresholds (rangeExpansion ≥ 1.5 AND directionalPersistence ≥ 0.65 for "breakout") are calibrated for equity swings, not Deriv synthetic spike behaviour. BOOM/CRASH spikes are directional but have irregular ATR ratios, so most fall through to "unknown" instead of the symbol-native family. strategyFamilyCandidate is always copied from moveType, so it is also mostly "unknown" for BOOM300/CRASH300.

Bug C — GPT-5.1 rejects max_tokens parameter Error: 'max_tokens' is not supported with this model. Use 'max_completion_tokens' instead. The chatComplete() wrapper passes through max_tokens unchanged. GPT-5.1 (and o-series models) require max_completion_tokens. Every AI call in the calibration passes, AI chat, research route, and openai.ts internal functions is affected.

Done looks like
Avg move % in the Move Calibration UI matches the raw export data numerically
BOOM300 move family distribution shows boom_expansion (not mostly unknown)
CRASH300 move family distribution shows crash_expansion
R_75 and R_100 continue using reversal / continuation / breakout
GPT-5.1 AI calls no longer fail on max_tokens
AI Analysis, calibration passes, and backtest AI chat all run successfully
TypeScript compiles without new errors

Out of scope
Redesigning the calibration system
Changing live trading logic or trade management
Chat UI redesign
Adding new symbols

NON-NEGOTIABLE RULES
movePct is stored in the DB as a decimal fraction (e.g., 0.0533 = 5.33%). Never change this storage convention.
avgMovePct and medianMovePct must use a single consistent unit across backend and UI — the simplest fix is to store them as fractions and let the UI do * 100 for display.
BOOM300 must map to boom_expansion, CRASH300 to crash_expansion — these values must be stored in strategyFamilyCandidate (text column, no constraint).
moveType may remain as the generic structural label (breakout/continuation/reversal/unknown) OR be updated to symbol-native values — executor's choice, but must be consistent with UI display.
Do NOT add max_tokens to gpt-5.1 calls anywhere. Use max_completion_tokens exclusively for gpt-5.x models.
Pre-existing TS errors (layout.tsx, data.tsx, decisions.tsx, diagnostics.tsx) must not be introduced as new errors by this task.

Remove max_tokens from the fallback path too if FALLBACK_MODEL also requires it.

Update contextRetriever.ts where it formats calibration profile avgMovePct / medianMovePct as display percentages (add * 100 there).

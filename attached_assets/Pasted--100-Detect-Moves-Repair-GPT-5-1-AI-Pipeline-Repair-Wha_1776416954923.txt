#100 — Detect Moves Repair + GPT-5.1 AI Pipeline Repair

## What & Why

Two major issues are now confirmed in the current Move Calibration flow:

### Problem 1 — Detect Moves summary is wrong / misleading

The BOOM300 Detect Moves export shows row-level `movePct` values as decimal fractions such as:

* `0.0533`
* `0.1067`
* `0.1578`

which clearly represent approximately:

* **5.33%**
* **10.68%**
* **15.78%**

But the UI summary is showing:

* **Avg move % = 698.4%**
* **By family = unknown 311, continuation 2**

This is incorrect.

The detector/export proves:

* average move formatting/aggregation is wrong
* BOOM300 family classification is wrong
* the move detector is still acting like a generic swing-leg finder instead of a symbol-native move finder

### Problem 2 — AI is still broken after GPT-5.1 migration

The app now fails with:

* `Unsupported parameter: 'max_tokens' is not supported with this model. Use 'max_completion_tokens' instead.`

This means the GPT-5.1 migration is incomplete:

* central AI wrapper exists
* model swap was attempted
* but the request payloads are still not valid for the active GPT-5.1 usage path

So this task is a focused repair pass for:

* Detect Moves correctness
* symbol-native family classification
* GPT-5.1 request compatibility
* end-to-end AI Analysis / Move Calibration pass execution

This is a repair task, not a broad redesign.

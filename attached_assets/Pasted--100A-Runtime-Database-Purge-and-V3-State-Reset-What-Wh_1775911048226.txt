#100A - Runtime Database Purge and V3 State Reset

## What & Why
Before rebuilding the frontend, the runtime/application database state must be cleaned so the new V3 UI does not inherit stale noise from prior versions.

The current database still contains mixed state from prior iterations:
- backtest history
- stale trades
- stale signal logs
- old platform state
- old settings / feature flags / runtime keys

This task is a **database runtime reset only**.

It must:
- purge stale runtime/application data
- preserve market history
- identify which settings/platform state keys are still actually used by V3
- reset the system into a clean V3 runtime state

No frontend rebuild in this task.

---

## Non-negotiable rules
- Do NOT touch or delete:
  - candles
  - ticks
  - spike_events
  - symbol market history
- Do NOT rebuild frontend in this task
- Do NOT change trading logic in this task
- Do NOT silently delete settings/state without first identifying whether V3 still uses them
- This reset must be repeatable for future rebuilds

---

## Required purge
Purge:
- backtest_runs
- backtest_trades
- signal_log
- trades

Also inspect and clean/reset:
- platform_state
- settings
- feature flags
- old runtime keys
- stale UI state if stored in DB

---

## Tasks

### 1. Audit current DB runtime/application state
Return a matrix for:
- table / key
- purpose
- currently used by V3? yes/no
- action: keep / delete / reset
- reason

At minimum include:
- backtest_runs
- backtest_trades
- signal_log
- trades
- platform_state
- settings-related tables/keys
- any persisted app/runtime flags

### 2. Purge stale runtime tables
Delete all rows from:
- backtest_runs
- backtest_trades
- signal_log
- trades

### 3. Clean platform_state
Delete/reset stale keys not used by current V3 runtime.
Keep only keys genuinely required by V3.

### 4. Clean settings state
Delete/reset settings/flags from prior iterations that are no longer used by V3.
Keep only currently active V3 settings.

### 5. Make reset repeatable
Create a safe rebuild/reset routine or script so this cleanup can be rerun in future.
It must be explicit and logged.

### 6. Verify preservation of market history
Prove that:
- candles remain intact
- ticks remain intact
- spike_events remain intact

### 7. Verify clean runtime state
After reset:
- no trades
- no signal logs
- no backtests
- no stale runtime junk
- system still boots cleanly

---

## Required deliverables
Return:
1. tables purged
2. platform_state keys deleted/reset/kept
3. settings keys deleted/reset/kept
4. reset script/command added
5. proof candles/ticks/spike_events were preserved
6. proof runtime state is clean
7. build/test/server result
8. explicit confirmation:
   - runtime DB state is clean
   - market history was preserved
   - no frontend changes were made

---

## Quality bar
This task succeeds only if the app can now start from a clean V3 runtime state with historical market data preserved.

#100C.1 - UI Surface Audit, Endpoint Mapping, and Ownership Inventory

## What & Why
The operational core UI is still inconsistent.

There are currently overlaps and unclear ownership between:
- Overview
- Decisions
- Trades
- Data
- Diagnostics

Some functionality appears in multiple places.
Some diagnostics surfaces are better than the newer operational pages.
Some controls exist but do not behave clearly.
Streaming appears in too many places.
It is not yet clear which page should own what.

Before making more changes, we need a precise inventory of the current UI and how it maps to the backend.

This task is an **audit only**.
Do NOT redesign.
Do NOT rebuild.
Do NOT refactor page ownership yet.
Do NOT remove sections yet unless absolutely necessary for the audit to run.

The goal is to produce a clean report of:
- what exists now
- which routes/pages/components own it
- which endpoints feed it
- which controls are duplicated
- which controls are broken / no-op / unclear
- which sections appear better suited to remain where they are
- which sections are redundant or misplaced

This report will then be used to create Task #100C.2 with precise changes.

---

## Non-negotiable rules
- Do NOT rebuild pages in this task
- Do NOT move sections between pages in this task
- Do NOT delete large areas of UI in this task
- Do NOT patch over issues instead of reporting them
- Do NOT make assumptions about desired final ownership
- Focus on accurate inventory and evidence

Minor instrumentation is allowed only if needed to identify ownership or endpoint usage clearly.

---

## Audit scope
Audit these current primary pages/routes:
- /overview
- /decisions
- /trades
- /data
- /diagnostics

Also note any relevant legacy/dev-only routes if they still expose overlapping content:
- /legacy/*
- redirects from /signals, /v3-backend, etc.

---

## Tasks

### 1. Build page-by-page UI inventory
For each audited page, list every major visible section/component.

For each section/component return:
- page route
- section name
- high-level purpose
- whether it is summary / control / diagnostic / operational / research / legacy-style
- whether it appears unique or duplicated elsewhere
- whether it appears useful / confusing / stale / broken / unclear

At minimum produce an inventory for:
- Overview
- Decisions
- Trades
- Data
- Diagnostics

---

### 2. Map each visible section to its backing endpoint(s)
For every major UI section, identify:
- exact backend endpoint(s) used
- whether data is live / cached / derived
- whether the endpoint is operational, diagnostic, research, export, or settings-related
- whether the section is showing the endpoint output cleanly or confusingly

Return a matrix:
- page
- section
- endpoint(s)
- data type
- quality note

---

### 3. Audit interactive controls
For every visible user control/button/toggle/input on those pages, return:

- page
- section
- control name
- intended action
- actual backend route/action triggered
- does it work? yes/no/unclear
- visible UX feedback on click? yes/no
- updates state after action? yes/no
- duplicate of another control elsewhere? yes/no
- notes

This is especially important for:
- streaming controls
- reconcile/top-up/enrich controls
- export controls
- any settings-like controls
- any action buttons in diagnostics or data

---

### 4. Audit duplication / overlap
Identify all duplicated or overlapping concepts across the pages.

Examples:
- streaming summary vs streaming control
- integrity summary vs diagnostics integrity detail
- export controls in multiple places
- data coverage shown in multiple places
- symbol state shown in multiple places
- decision metrics repeated elsewhere
- runtime information repeated elsewhere

For each overlap, return:
- concept
- pages where it appears
- whether duplication is good, acceptable, or confusing
- whether one instance is clearly better than another

Do NOT decide final ownership yet — just report clearly.

---

### 5. Identify broken / misleading / no-op UX
List any UI elements that:
- do nothing
- produce unclear results
- do not visibly update after action
- misrepresent backend truth
- show stale or duplicated information
- should probably not exist in their current form

Especially check:
- streaming buttons
- date inputs
- export controls
- diagnostics actions
- page summaries that contradict detail views

---

### 6. Compare “better diagnostic implementation” vs “weaker operational implementation”
You specifically noted that some Diagnostics content is better than what was built into the other tabs.

For every such case, identify:
- the concept
- the page where the better version exists
- the page where the weaker version exists
- what exactly makes one better:
  - clearer layout
  - better numbers
  - better endpoint truth
  - better interaction
  - less duplication
  - better drilldown
  - etc.

This is important for writing 100C.2 later.

---

### 7. Identify current page ownership candidates
Without making final changes, provide a recommendation matrix for each concept:

- concept
- current page(s)
- likely best owner
- likely summary-only pages
- likely pages where it should be removed

Important:
These are recommendations only for planning 100C.2.
Do NOT implement the moves yet.

Key concepts to classify:
- streaming control
- streaming status
- symbol state
- data coverage
- candle integrity
- export readiness
- export execution
- research ranking
- AI report
- trade lifecycle
- decision reasoning
- diagnostics/runtime detail
- settings controls

---

### 8. Capture route and component ownership
Return which files currently own each audited page and major section.
This should include:
- route
- page file
- major child components if relevant
- whether it is fresh V3 / provisional / legacy-derived / unclear

We need to know exactly what is still surviving from prior iterations.

---

### 9. Produce a final audit summary for planning 100C.2
At the end, provide a concise but complete summary with:

1. what is currently working well
2. what is duplicated
3. what is confusing
4. what is broken
5. which screens are strongest today
6. which screens are weakest today
7. which concepts clearly need a single owner
8. what 100C.2 should focus on first

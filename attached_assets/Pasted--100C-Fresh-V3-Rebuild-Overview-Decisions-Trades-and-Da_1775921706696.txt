#100C - Fresh V3 Rebuild: Overview, Decisions, Trades, and Data

## What & Why
Task #100B completed the shell/route reset, but it did NOT yet complete the actual fresh V3 operational rebuild.

We now have:
- legacy V2 pages retired from main ownership
- fresh scaffold pages for some routes
- but some important primary pages still come from the prior Task #99 patch era
- and we do NOT want those provisional/patch pages to become the final V3 experience

This task builds the first half of the **real fresh V3 operational UI** from scratch.

It covers the operational core pages:
- Overview
- Decisions
- Trades
- Data

These pages must become:
- fresh V3 pages
- data rich
- explanation first
- driven only by working backend truths
- free from V2 legacy assumptions
- free from Task #99 provisional patch ownership

This is not a cosmetic polish pass.
This is the first real content rebuild after the shell reset.

---

## Critical override from prior tasks
The following rules override any temptation to reuse existing page content:

### 1. Do NOT reuse kept Task #99 page implementations as final page content
Any page/component currently surviving from Task #99 must be treated as **provisional only**, not final ownership.

That means:
- `/decisions`
- `/data`
- `/help`
- `/diagnostics`

must NOT be treated as already “good enough” just because they exist.

In this task specifically:
- Overview
- Decisions
- Trades
- Data

must be rebuilt as **fresh V3 pages from scratch**, even if routes/components already exist.

### 2. No patching old tables/components into “new” pages
Do NOT:
- take an old grid/table/card and rename labels
- keep old layout ownership and swap data sources
- preserve V2/V1 composition under new headings

Build fresh page structures and fresh page-level ownership.

### 3. Fresh means new page ownership
Each of the 4 pages in this task must have:
- fresh page-level structure
- fresh page sections
- fresh ownership of layout/content composition

Reusable low-level UI primitives are fine if generic.
Legacy page composition is not fine.

### 4. Only show backend truths that actually exist
Do NOT invent fields.
Do NOT show fake placeholders.
If something is not backed by current backend truth:
- omit it
- or add minimal backend support only if essential and small

### 5. Explanation-first is mandatory
This user wants to understand:
- what triggered
- what failed
- why it failed
- what is expected
- what the system is doing now

These pages must show reasoning, not just status badges.

## Required pages

### 1. Overview page — build from scratch
This page must answer:
- what is the system doing right now?
- what matters most right now?
- what is healthy or broken right now?

At minimum show:
- current mode
- active trading symbols
- streaming summary
- last scan time
- last trade time
- open trades count
- decision counts by state
- active warnings/blockers
- capital summary if available
- current paper/demo/live summary if available
- data health summary for active symbols
- top current opportunities if backed by backend
- latest research freshness indicator if backed by backend

Recommended sections:
- System Status
- Trading Activity
- Data Health
- Current Opportunities
- Warnings / Attention Needed

Rules:
- use real numbers
- avoid generic filler cards
- no old dashboard ownership reused

---

### 2. Decisions page — build from scratch
This page is critical.

It must show the actual V3 decision flow clearly.

For each decision row/card show where available:
- symbol
- engine name
- direction
- decision state:
  - approved
  - blocked
  - traded
  - pending
  - rejected
  - suppressed
- composite/confidence score
- expected move %
- expected hold profile
- timestamp
- AI verdict if real and current
- coordinator outcome
- allocator outcome

Most importantly, show explanation fields:
- why it passed
- why it failed
- why it was blocked
- threshold misses
- gating failures
- conflict with stronger candidate
- exposure constraint
- allocator rejection
- data quality issue if relevant

Must include:
- filters for symbol / engine / state / date range
- detail panel / drawer / drilldown for a selected decision
- export if already supported, but not as the main focus

The page must make it obvious why a decision did or did not become a trade.

This must be a fresh page, not the old Signals logic in disguise.

---

### 3. Trades page — build from scratch
This page must explain live and historical trade behavior in a V3-native way.

For each trade show where available:
- symbol
- engine origin
- direction
- entry timestamp
- entry price
- capital allocated
- quantity/size if available
- TP
- SL
- trailing stop status
- current stage
- expected move %
- expected hold profile
- actual hold time
- current PnL
- realized PnL if closed
- exit reason when closed

Must also show rationale:
- why the trade was opened
- what it expected to capture
- current status vs expectation
- what invalidates it
- why it exited

Recommended sections/tabs:
- Open Trades
- Closed Trades
- Trade Lifecycle / Stage Transitions
- Trade Attribution by Symbol/Engine

This page must not carry old trade-screen assumptions forward.

---

### 4. Data page — build from scratch
This page must become the true operational data console.

It must show:
- all 28 symbols
- per-symbol state:
  - streaming
  - available
  - idle
  - disabled
- active trading symbols clearly highlighted
- per-symbol data coverage
- per-timeframe availability
- real vs interpolated counts
- first/last dates
- last reconcile / repair / enrich if available
- current export health if available

Recommended tabs/sections:
- Symbol State
- Candles
- Live Ticks
- Spike Events
- Integrity / Coverage Summary

It must support where backend allows:
- per-symbol stream/pause controls
- reconcile/top-up action
- enrich action
- inspect coverage details

The Data page must make it obvious:
- what data exists
- what is live
- what is repaired
- what is interpolated
- what is healthy

This must be a fresh page, not a repackaged old data/admin screen.

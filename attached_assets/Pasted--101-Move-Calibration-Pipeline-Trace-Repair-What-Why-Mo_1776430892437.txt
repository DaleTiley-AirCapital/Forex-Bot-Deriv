## Tasks

### 1. Trace the full BOOM300 calibration pipeline
Trace the complete data path for BOOM300 through these stages:

1. Detect Moves
2. stored detected_moves rows
3. pass-run creation
4. precursor/trigger/behavior/extraction pass execution
5. pass result persistence
6. calibration aggregation
7. strategy_calibration_profiles write
8. profile fetch route
9. profile export route

For each stage, confirm:
- what input it received
- what output it wrote
- what filters were applied
- whether BOOM300 rows were dropped there

This must be done using the actual current repo code and live data path, not assumptions.

### 2. Identify the exact BOOM300 drop-out point
Find exactly where BOOM300 is being lost.

Examples of acceptable root causes:
- family mismatch
- wrong strategy key
- profile query filtering
- aggregator grouping mismatch
- pass results not persisted
- export route filtering wrong field
- symbol-native family value not matching expected profile strategy
- any equivalent real cause

You must identify the exact code path and exact reason.

### 3. Audit BOOM300 family key consistency across pipeline
Inspect all BOOM300-relevant fields across the calibration path, including:
- `symbol`
- `moveType`
- `strategyFamilyCandidate`
- any pass `strategyFamily`
- profile grouping keys
- route params used by `/profile/:symbol/:strategy`
- export grouping/filter logic

Confirm whether BOOM300 is consistently represented as:
- `boom_expansion`
- or another key
- or generic `breakout/continuation/reversal/unknown`

Fix any inconsistency so the full pipeline agrees on the same BOOM300 strategy key.

### 4. Audit profile aggregation preconditions
Inspect the profile aggregator and determine:
- what minimum data it requires before writing a profile
- whether BOOM300 currently fails those preconditions
- whether that failure is valid or due to a bug

If a valid precondition exists, make it observable in logs/response.
If the precondition logic is wrong for BOOM300, fix it.

### 5. Audit pass persistence
Confirm that each calibration pass for BOOM300:
- executes
- returns usable data
- writes records to the expected tables
- is linked to the correct move/profile grouping keys

If pass results are generated but not persisted or not retrievable by the aggregator, fix that.

### 6. Audit profile route and export route filtering
Inspect:
- `GET /api/calibration/profile/:symbol/:strategy`
- `GET /api/calibration/export/:symbol`

Confirm:
- whether they are filtering on the correct family key
- whether they are expecting a strategy value that BOOM300 profiles never use
- whether export is excluding valid profiles unintentionally

Fix any mismatch.

### 7. Add explicit calibration-stage diagnostics
Add lightweight diagnostics/logging so future failures are visible.

At minimum log or expose:
- detected move count
- pass run count
- pass success/failure counts
- aggregation candidate count
- profile rows written
- profile rows exported

Do not spam logs, but make the drop-out stage visible.

### 8. Validate BOOM300 end to end
Run:
- Detect Moves
- Run All Passes
- Full Calibration
- Profile fetch
- Export

Confirm the complete path now works for BOOM300.

### 9. Validate other active symbols for regressions
Quickly confirm the same repair did not break:
- CRASH300

This is a regression check only, not a full redesign.

#103 - Candidate Lifecycle, Adaptive Scan Cadence, Rolling Window Memory, and Decision Log De-Duplication

## What & Why
The native engine scoring rebuild is complete across all 4 active symbols, but the live system is still noisy and wasteful because it scans too frequently and logs the same blocked candidate state over and over.

Current problem:
- scheduler baseline is effectively 60-second decision scanning
- the same symbol/engine/direction candidate can be logged repeatedly for long periods
- this burns compute
- this floods the decision log with near-identical blocked states
- it makes it hard to tell whether the engine is improving toward a tradeable setup or just stuck in noise

This task must fix that.

The system should:
- scan all active symbols on a slower baseline cadence
- maintain rolling candidate memory per symbol/engine/direction
- escalate promising candidates into higher-frequency watch mode
- only log meaningful state transitions or material score changes
- keep full data streaming and candle building intact
- preserve visibility into near-miss candidates without spamming identical log entries

This is NOT a scoring redesign task.
This is NOT a threshold recalibration task.
This is a runtime decision-lifecycle and compute-efficiency task.

## Objective

1. reduces compute usage
2. reduces decision-log spam

4. supports rolling window memory

## Core behavioral model to implement

### Baseline decision scan cadence
All active symbols should perform a full decision evaluation every:
- 5 minutes

### Watch mode
If a candidate becomes promising, that specific candidate can be promoted into:
- watch mode

Watch mode means:
- more frequent evaluation
- finer-grained monitoring
- but only for that candidate
- not for all symbols

### Tradeable state
A candidate becomes tradeable only when:
- it passes engine-native gate
- it passes allocator/mode gate
- the rolling setup memory still supports the move
- it is not just a brief noisy spike in score

### Candidate lifecycle states
Implement an explicit lifecycle state machine for each candidate:

- `idle`
- `watch`
- `qualified`
- `tradeable`
- `executed`
- `expired`

Definitions:

#### idle
No candidate worth tracking.

#### watch
Candidate has become interesting enough to monitor more closely, but is not yet good enough to trade.

#### qualified
Candidate has passed the engine-native gate or is very close and structurally improving.

#### tradeable
Candidate passes engine-native gate and allocator/mode requirements and is valid for execution.

#### executed
Trade opened.

#### expired
Candidate was previously watch/qualified but has degraded or become invalid.

## Candidate identity model
A candidate must be keyed by at least:

- symbol
- engine name
- direction
- setup family or equivalent native setup identity

This is required so the system can distinguish:
- a continuing watch candidate
from
- a truly new candidate

## Rolling window memory requirements

### 1. Maintain rolling candidate memory
For each tracked candidate, maintain a rolling memory object that stores:

- current lifecycle state
- first seen timestamp
- last seen timestamp
- last score
- best score seen
- current breakdown
- previous breakdown
- engine gate status
- allocator/mode gate status
- latest rejection reason
- regime and regime confidence
- count of consecutive improving scans
- count of consecutive degrading scans

### 2. Rolling memory must update incrementally
As new data comes in:
- old state is updated
- not replaced blindly
- score evolution can be measured
- breakdown evolution can be measured

### 3. Candidate quality trajectory must be visible
We need to know whether a candidate is:
- improving
- flat
- degrading
- peaking

This is important because execution should happen when the setup becomes genuinely strong, not just when it first appears.

## T002 — Add per-candidate watch mode
Implement a watch-mode scheduler path.

### Watch trigger
A candidate should be promoted from idle to watch when one of the following is true:

- native score crosses a configurable watch threshold
- OR candidate is within a small band below engine-native gate and improving
- OR breakdown structure strongly suggests setup maturation

Starting watch-threshold guidance:
- use something like native >= 60 for watch eligibility
- but allow per-engine override if needed
- if you refine this, report the exact rule in final output

### Watch cadence
Once in watch mode, the candidate should be re-evaluated more frequently than baseline.

Use:
- 1-minute evaluation cadence for watched candidates

### Important
Do NOT globally rescan all symbols every minute.
Only the watched candidate(s) should receive higher-frequency reevaluation.

## T003 — Add candidate lifecycle state machine
Implement the lifecycle state model described above.

Required transitions include at minimum:

- `idle -> watch`
- `watch -> qualified`
- `qualified -> tradeable`
- `tradeable -> executed`
- `watch/qualified -> expired`
- `qualified -> watch` if setup degrades but still remains interesting

State transitions must be explicit and inspectable.

## T004 — Add material-change rules for logging
Decision log entries must no longer be emitted every scan if nothing meaningful changed.

### Only log on meaningful events
Log an entry when any of the following occurs:

1. Candidate first enters `watch`
2. Candidate score changes by a meaningful amount
3. Candidate’s weak components materially change
4. Candidate crosses engine-native gate
5. Candidate crosses allocator/mode gate
6. Candidate becomes tradeable
7. Candidate is executed
8. Candidate expires
9. Candidate direction flips or engine winner changes materially

### Minimum score delta rule
Implement a minimum score-change threshold before re-logging.

Recommended starting rule:
- re-log only if native score changes by >= 4 points

If you choose a different value, explain why.

### Weak-component change rule
If the weak components changed materially, re-log even if score delta is smaller.

Examples:
- pullbackQuality improves from 5 -> 35
- expansionQuality improves from 11 -> 45
- entryEfficiency changes enough to alter execution plausibility

### No duplicate spam
If:
- same candidate
- same lifecycle state
- same rejection reason class
- same weak components
- score delta below threshold

then do NOT insert a new signal_log row.

The goal is that the decision log becomes a real lifecycle log, not a spam stream of repeated blocked entries.

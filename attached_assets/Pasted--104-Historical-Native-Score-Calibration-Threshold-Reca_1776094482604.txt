#104 - Historical Native Score Calibration, Threshold Recalibration, and Ideal-Setup Normalization

## What & Why
Tasks #101A–D completed engine-native scoring for all 4 active symbols.
Task #103 added lifecycle, deduplication, and slower baseline scanning.

We are now at the point where the remaining major blocker is:
- native score calibration
- mode threshold calibration
- score scaling truth

Right now the current secondary thresholds are:
- Paper = 85
- Demo = 90
- Real = 92

These thresholds were inherited and preserved during migration, but they were NOT recalibrated against the new symbol-native scoring systems.

That means the system may still be suppressing nearly all otherwise-valid candidates.

This task must calibrate the current native scoring systems against historical target moves and then set the correct thresholds from evidence.

This task is not a vague tuning pass.
It is a structured calibration task.

---

## Core calibration philosophy (non-negotiable)
For each symbol and engine:

- the ideal historical setup the engine is designed to capture should score near 100
- strong but imperfect setups should score below that
- weaker/borderline setups should score meaningfully lower
- noise should stay low

Do NOT treat the current raw score scales as automatically correct.

This task must verify whether the score scale itself is properly normalized.

If ideal historical setups are only scoring 60–75, that is a calibration failure.
If ideal textbook setups cluster near 90–100, that is a healthy score scale.

---

## Non-negotiable rules

### 1. Use historical target moves, not random examples
Do not calibrate on random candles.
Calibrate on the actual move types each engine is meant to capture.

### 2. Calibrate per symbol and per engine
No universal one-size-fits-all calibration.

This task must analyze all active live engines:
- BOOM300
- CRASH300
- R_75 reversal / continuation / breakout
- R_100 reversal / breakout / continuation

### 3. Distinguish score scaling from threshold selection
Two different questions must be answered:

1. Is the score scale correct?
   - do ideal setups score near 100?

2. What thresholds should be used?
   - Paper
   - Demo
   - Real

Do not conflate those.

### 4. Preserve long-hold philosophy
Do NOT optimize for micro-move scalping.
Do NOT lower standards just to make the system fire more often.
The goal is:
- high-conviction
- long-hold
- cash extraction
- realistic trade frequency

### 5. No docs rewrite in this task
Keep scope tight:
- scoring calibration
- threshold calibration
- minimal UI/observability additions if needed
- no broad refactor

### 6. No fallback logic
If a native score formula is badly scaled, fix it.
Do not add hacks on top.

---

## Objective
Use historical data and the current engine logic to:
1. identify the real score distributions per engine
2. test whether ideal setups score near 100
3. recalibrate score scaling if needed
4. derive evidence-based thresholds for Paper / Demo / Real
5. produce a calibration report that justifies the new values

## T001 — Build calibration runner
Create a calibration runner that replays historical data through the current native engine scoring logic.

For each engine and symbol, the runner must record:
- timestamp
- symbol
- engine name
- direction
- native score
- component breakdown
- engine gate pass/fail
- allocator threshold pass/fail at multiple score levels
- eventual realized move after the candidate
- realized hold profile
- drawdown before favorable move
- whether the move qualifies as a target move for that engine

This can be:
- a dedicated calibration script
- or a research endpoint/job
- or a reusable backend analysis module

But it must be runnable and reproducible.

## T004 — Test “ideal setup scores near 100”
For each engine:
- identify strong historical target examples
- inspect what native scores they actually receive
- determine whether textbook setups cluster near 90–100

If not, diagnose why.

Possible reasons:
- score components underweighted
- expectedMoveSufficiency scaling too harsh
- entryEfficiency too punitive
- structural quality scale too compressed
- all good setups capped in the 60–75 range

This is the most important part of the task.

### Requirement
Do not just report “threshold too high.”
First determine whether the score scale itself is badly normalized.

---

## T005 — Normalize score scaling if ideal setups are under-scored
If the calibration shows that true ideal historical setups are not scoring near the top of the range, recalibrate the score normalization.

Allowed adjustments:
- component scaling curves
- breakpoint maps
- normalization constants
- weighting refinements
- engine-native gate refinement if strongly justified

Not allowed:
- arbitrary hacks to simply force more trades
- lowering thresholds without diagnosing score-scale problems first

### Goal
After recalibration:
- ideal historical setups should score near 90–100
- strong but imperfect setups below that
- weak/noisy setups materially lower

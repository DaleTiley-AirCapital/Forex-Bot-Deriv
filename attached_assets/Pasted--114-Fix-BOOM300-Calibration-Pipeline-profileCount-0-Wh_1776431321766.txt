#114 - Fix BOOM300 Calibration Pipeline (profileCount: 0)

What & Why
Move Calibration runs to completion for BOOM300 but exports show profileCount: 0, profiles: []. The calibration pipeline has a confirmed break point that prevents extraction pass results from being written for BOOM300 and CRASH300. The failure is traced and the exact drop-out point is identified — this task fixes it.

Root cause (confirmed via code trace): calibrationPassRunner.ts:170: if (moveType) conditions.push(eq(detectedMovesTable.moveType, moveType))

When the UI selects strategyFamily = "boom_expansion", it calls toBackendMoveType("boom_expansion") which returns the string "all". Because "all" is truthy, the run-passes route sets resolvedMoveType = "all" and the pass runner adds WHERE move_type = 'all' to the DB query. No moves have move_type = 'all' (valid values are "breakout"/"continuation"/"reversal"/"unknown"), so filteredMoves = []. The extraction pass hits if (moves.length === 0) return; at line 40 and immediately returns without writing any profiles. Per-move passes also find nothing to process.

Cascade: strategy_calibration_profiles has 0 rows for BOOM300 → GET /api/calibration/export/BOOM300?type=profile → profileCount: 0, profiles: [].

Same bug affects CRASH300 via "crash_expansion" → toBackendMoveType → "all".

Secondary issue discovered: calibrationPassRunner.ts does not route boom_expansion/crash_expansion through strategyFamilyCandidate (as getDetectedMoves does after Task #106) — inconsistency that would surface if a filter other than "all" is used.

No other code is broken. The pipeline stages before (detect moves) and after (profile read/export routes) are structurally correct. Only the move-query in the pass runner is wrong.

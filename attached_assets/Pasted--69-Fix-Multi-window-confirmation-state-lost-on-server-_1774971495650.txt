#69 - Fix: Multi-window confirmation state lost on server restart
Problem
The multi-window confirmation system requires a signal to appear in 2 separate 1-hour windows before it is promoted. However, the confirmation state is stored in-memory only (a JavaScript Map in pendingSignals.ts). Every Railway deploy or server restart clears this Map completely.

Result: CRASH300 mean_reversion buy signal was detected in 3 consecutive hourly windows (12:31, 13:24, 14:24) but each time shows "1/2 windows" — because the server restarted between each detection (from Tasks #65, #66, #67 deploys), resetting confirmCount to 1 each time. A strong 82-score setup that should have been promoted in 2 hours took 3+ hours and still hasn't confirmed.

Fix
Persist the pending confirmation state to the existing PostgreSQL database.

DB changes
Add a pending_confirmations table:

key (text, PK) — the signal key: symbol|family|direction|mode
symbol, strategy_name, strategy_family, direction, mode (text)
confirm_count (int)
required_confirmations (int)
first_detected_at, last_confirmed_at (bigint — Unix ms)
last_composite_score, last_expected_value, last_score (float)
window_timestamps (text — JSON array of bigints)
price_at_first, price_at_last (float)
promoted (boolean)
pyramid_level (int)
Add the table to lib/db/src/schema.ts and wire it into initDb() in lib/db/src/index.ts.

pendingSignals.ts changes
Convert the in-memory Map to a write-through cache backed by the DB:

loadPendingSignalsFromDb() — called once at startup by the scheduler to hydrate the in-memory Map from DB rows
confirmSignal() — after writing to the in-memory Map, upsert the row to DB
removePendingSignal() / expireStaleSignals() / invalidateUnconfirmedPending() — after deleting from Map, delete from DB
The DB instance must be passed in (or imported) in pendingSignals.ts
Scheduler startup change
Call loadPendingSignalsFromDb() once at scheduler init time, before the first scan.

Files
lib/db/src/schema.ts — add pendingConfirmationsTable
lib/db/src/index.ts — add CREATE TABLE in initDb(), add Drizzle export
artifacts/api-server/src/lib/pendingSignals.ts — add DB persistence (write-through)
artifacts/api-server/src/lib/scheduler.ts — call loadPendingSignalsFromDb() at init
Done Looks Like
After a Railway redeploy, the server restores all in-progress confirmations from DB on startup
A CRASH300 signal detected in window 1 (before restart) correctly shows confirmCount=1 after restart
The signal promotes in window 2 instead of being reset to 1 each time
Promoted signals are cleaned from DB after promotion
Stale signals (4h gap) are still expired and removed from DB

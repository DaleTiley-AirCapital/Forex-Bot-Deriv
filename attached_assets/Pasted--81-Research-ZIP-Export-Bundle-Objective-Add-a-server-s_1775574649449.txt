#81 - Research ZIP Export Bundle

## Objective

Add a server-side research export endpoint that accepts a symbol, timeframe, date range, and optional chunk size, then streams back a single ZIP file containing raw candle data plus integrity metadata.

This is an export-layer addition only.

## Critical scope constraints

Do NOT modify:

* trading logic
* signal logic
* strategy logic
* scoring
* backtest engine
* scheduler
* existing execution flow
* unrelated routes
* DB schema unless absolutely required for export-only reasons

Do NOT refactor unrelated code.

Only add the minimum code required to support this research export capability.

## Endpoint

Add:

POST /api/export/research

Request body:

* symbol
* timeframe
* startDate
* endDate
* maxCandlesPerChunk? (default 25000, max 50000)

Optional query param:

* csv=true to include matching CSV chunk files in the ZIP

## Response

Stream a single ZIP file directly in the HTTP response.

ZIP filename format:
{SYMBOL}*research_pack*{YYYY-MM-DD}*to*{YYYY-MM-DD}.zip

Example:
BOOM300_research_pack_2025-09-01_to_2025-10-31.zip

No loose files should be written to disk.

Important:
Do not assemble the entire export in memory before sending.
Generate and stream ZIP entries incrementally.

## ZIP contents

The ZIP must always contain:

* manifest.json
* validation.json
* one or more raw candle JSON chunk files

Optional:

* matching CSV chunk files if csv=true

## Chunking rules

Chunking is internal only.

Rules:

1. Split by month first
2. If a month exceeds maxCandlesPerChunk, split further into fixed-size chunks
3. Default chunk size = 25000 candles
4. Maximum allowed chunk size = 50000 candles

Chunk filename format:

* BOOM300_2025-09_chunk_001.json
* BOOM300_2025-09_chunk_002.json
* BOOM300_2025-10_chunk_001.json

## Candle export requirements

Read candles from backgroundDb, not the main trading pool.

Query requirements:

* explicit ORDER BY timestamp ASC
* deterministic batching
* batch DB reads in 10000-row increments
* do not fetch the whole export into memory at once if avoidable

Preserve raw candle values exactly as stored.

Each exported candle must keep the existing raw fields exactly as stored:

* timestamp
* open
* high
* low
* close
* volume OR tickCount, depending on source

Rules:

* do not alter numeric precision
* do not smooth data
* do not fill gaps
* do not deduplicate silently
* do not reorder candles beyond explicit ascending timestamp order
* do not add strategy labels, signals, or classifications

## JSON chunk structure

Each chunk JSON file must contain:

* symbol
* timeframe
* exportDate
* chunkIndex
* totalChunks
* candleCount
* firstTimestamp
* lastTimestamp
* candles: [ ... ]

## Manifest

manifest.json must contain:

* symbol
* timeframe
* exportCreatedAt
* sourceStartTimestamp
* sourceEndTimestamp
* totalCandles
* totalChunks
* chunkFiles: [
  {
  fileName,
  candleCount,
  firstTimestamp,
  lastTimestamp
  }
  ]

## Validation

validation.json must contain:

* symbol
* timeframe
* totalCandlesExpected
* totalCandlesExported
* duplicateTimestampCount
* missingIntervalCount
* timestampsStrictlyAscending
* firstTimestamp
* lastTimestamp
* validationPassed
* notes

Validation rules:

1. Check candle count exported vs source count
2. Check timestamps are strictly ascending
3. Check duplicate timestamps
4. Check missing intervals based on requested timeframe
5. Missing intervals are reported only, never filled
6. Duplicate timestamps are reported only, never silently removed
7. If validation fails, record it clearly in validation.json

#84 - Repository Structural Alignment to Guardian Skill

What & Why
Task #83 completed dead-code deletion and UI/backend parity fixes, but it did not perform the broader repository structure alignment required by the deriv-trading-repo-guardian skill.

This task must now reorganize the codebase into the intended responsibility-based structure so future changes are simple, modular, and do not require editing many unrelated files.

The goal is not to change trading behavior. The goal is to sort the code into the correct architectural ownership model and remove remaining mixed-responsibility structure.

Done looks like
- the repo is organized according to the guardian skill direction:
  - core
  - runtimes
  - infrastructure
  - ui
  - app
- each concern has a clear owner file/module
- trading logic ownership is explicit
- runtime-specific wrappers are separated from shared engine logic
- infrastructure concerns are separated from business logic
- UI files are separated from app wiring
- superseded files are deleted after migration
- no fallback or duplicate paths remain
- imports are updated cleanly
- TypeScript compiles with zero errors
- a before/after file map is included in the completion report

Out of scope
- no strategy behavior changes
- no changes to TP/SL/scoring thresholds unless required only to relocate ownership without changing behavior
- no DB schema changes
- no new features

Tasks

Phase 1 — Structural ownership audit
- list current top-level and major subfolder structure
- identify mixed-responsibility files
- identify which files belong in:
  - core
  - runtimes
  - infrastructure
  - ui
  - app
- identify any remaining duplicate ownership

Phase 2 — Canonical structure plan
- produce a target tree for the current repo
- map existing files to target destinations
- identify files that will be deleted after migration

Phase 3 — Re-home files without changing behavior
- move shared trading logic into core ownership
- move runtime wrappers into runtimes
- move data/broker/db/logging/stream plumbing into infrastructure
- keep UI screens/components under ui
- keep app wiring/routes/dependency composition under app
- do not duplicate files during migration

Phase 4 — Delete superseded files
- after imports are updated, delete replaced or obsolete files
- do not leave compatibility shims
- do not comment out old paths

Phase 5 — Validate
- run TypeScript check
- confirm app starts
- confirm server starts
- produce:
  - before/after file map
  - moved files list
  - deleted files list
  - canonical owner map
  - remaining technical debt

#86 - Repository Structural Alignment (API server lib/ split into core / infrastructure / runtimes)

## What & Why
The guardian skill mandates a responsibility-based repository layout:

- `core/` for trading engine and business logic
- `infrastructure/` for broker / OpenAI / export / scheduler / symbol validation / stream plumbing
- `runtimes/` for execution runners such as backtest/research simulation
- `ui/` and `app/` are already represented elsewhere and are out of scope for this task

The API server still keeps its main business and service files in a flat `src/lib/` directory, which hides ownership and makes future changes harder to reason about. This task performs a structural alignment only: move the existing API server `lib/` files into the correct responsibility-based directories and update all imports accordingly.

This is a file-ownership and import-path task only.
It is **not** a trading-behavior task.

## Non-negotiable rule
Never preserve old behavior through fallback logic. When behavior is intentionally changed, remove the previous path completely and update all callers to the new path.

Also:
- do not leave compatibility re-export stubs in `src/lib/`
- do not comment out the old imports
- do not preserve parallel old/new paths
- once all imports are updated, delete the old `src/lib/` directory entirely
- do not leave empty directories, shim files, wrappers, aliases, or temporary bridges behind

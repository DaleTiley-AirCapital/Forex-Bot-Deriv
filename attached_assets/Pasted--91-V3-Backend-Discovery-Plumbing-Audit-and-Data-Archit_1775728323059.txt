#91 - V3 Backend Discovery, Plumbing Audit, and Data Architecture Report

## What & Why
The repo has undergone major architectural changes toward V3, but the current UI and some backend behavior suggest that:
- parts of V2 are still visible or partially wired
- data storage behavior is not fully understood
- derived candle generation and historic/live continuity are not clearly verified
- research/export behavior may not yet match the actual database state
- several settings and controls may be redundant, stale, misleading, or not actually wired

Before making further backend or UI changes, we need a complete, code-backed inventory of how the system currently works.

This task is a **discovery, audit, and reporting task only**.

It must produce a precise report of:
1. how data is stored now
2. how live data and historic data interact
3. what timeframes exist, where they come from, and whether they are persisted or derived
4. what parts of the V3 backend are actually wired
5. what parts of the UI are stale / legacy / misleading / unwired
6. what must change next in backend before the UI is rebuilt

This task must not do the actual redesign yet.
It must gather everything needed so the next task can safely implement backend changes.

---

- Do NOT redesign the system yet
- Do NOT change trading logic yet

## Core questions this task must answer

### A. Data architecture
1. Where are live ticks stored?
2. Where are candles stored?
3. Where are spike/events stored?
4. Are historic downloads stored in the same tables as live-derived data?
5. Is there one unified time series per symbol/timeframe, or separate pathways?
6. What exact tables and columns are involved?
7. What indexes/constraints protect candle integrity?

### B. Timeframes
1. Which candle timeframes are currently persisted in the database?
2. Which are only derived in memory or on demand?
3. Which are used by live trading?
4. Which are used by research?
5. Which are exposed in the UI?
6. Are higher timeframes actually being built continuously from live data?
7. If historic backfill currently downloads only 1m and 5m candles, how are the others created?
8. Is there an existing function that can build missing derived candles from stored lower timeframe data?

### C. Historic vs live continuity
1. Startup/backfill says it downloads historic data — what exactly is downloaded now?
2. Does it really fetch 12 months only, or whatever the endpoint makes available?
3. What happens if more historical data is available from the API?
4. What happens if there are missing intervals/gaps in local storage?
5. Is there already a top-up or gap-fill mechanism?
6. If streaming was off for a while, what exactly happens when it resumes?
7. Does the system currently detect missing timestamps and refill them?
8. If not, where would that logic need to be inserted?

### D. Export integrity
1. What exactly does the current ZIP research export read from?
2. Does it export raw stored candles only, or transformed/derived data?
3. Does it validate continuity?
4. Does it preserve all timestamps and values?
5. Could it export incomplete or dirty data today?
6. If source data is dirty, is there any existing cleanup pass before export?
7. What exact backend modules and routes own export behavior now?

### E. AI integration
1. What AI endpoints/functions currently exist?
2. Is AI currently wired into live trade verification anywhere?
3. Is AI currently used in research/backtests?
4. What can AI currently access from the database?
5. What current AI usage is stale / no longer aligned to V3?
6. What backend pieces would be needed for AI to run independent research on stored market data and produce symbol-level findings?

### F. Frontend wiring audit
For each current page/tab, determine whether it is:
- fully wired and relevant to V3
- partially wired / misleading
- legacy/stale
- redundant
- using old V2 concepts

Audit these pages at minimum:
- Overview
- Research
- Signals
- Trades
- Data
- Settings
- Help

For each page, identify:
- what backend endpoints/hooks it calls
- whether those endpoints still match V3
- which displayed fields are legacy or misleading
- which controls/settings are not actually wired
- which tabs should be retired versus rebuilt

### G. Current V3 plumbing audit
Verify end to end:
1. scheduler → V3 scan path
2. V3 router → symbol engines
3. symbol coordinator
4. allocator
5. hybrid trade manager
6. execution handoff
7. logging/diagnostics

For each layer, report:
- module/file ownership
- whether it is active in runtime
- whether it is fully wired
- whether any legacy V2 logic still leaks into runtime or UI

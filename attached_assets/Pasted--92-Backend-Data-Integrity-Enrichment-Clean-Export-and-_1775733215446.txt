#92 - Backend Data Integrity, Enrichment, Clean Export, and AI Research Foundation

## What & Why
Task #91 discovery confirmed that the biggest problem is not trading logic — it is the backend data foundation.

The audit established that:
- data integrity is currently broken / inconsistent
- research/backtests/AI can be invalidated by destructive pruning and incomplete history
- the live system is effectively operating on 1m-only structure in places where multi-timeframe logic is expected
- AI evaluation is currently misleading in at least some flows
- export integrity is not guaranteed
- the UI must NOT be rebuilt yet until the backend truth is fixed first :contentReference[oaicite:0]{index=0}

This task is the backend foundation task that must happen before the full V3 UI rebuild.

This task must:
1. stop destructive data behavior
2. unify historic + live data into one continuous market history
3. build deterministic multi-timeframe candle enrichment
4. detect and repair missing timestamps / gaps
5. make research export safe and clean
6. correct AI plumbing so it can be used properly for research
7. prepare the backend foundation needed for the later V3 UI task

This is a backend-only task.
Do NOT attempt the UI overhaul in this task.

---

## Non-negotiable rules

### 1. No destructive pruning of valid history
If the system currently deletes candle history beyond 12 months or otherwise discards valid available data during research/backfill/export flows, that behavior must be removed or replaced.

Do NOT destroy valid historical data as part of normal runtime or research workflows.

### 2. No silent fallback logic
If data cleanup, enrichment, export validation, or AI research preparation fails:
- fail loudly
- log clearly
- surface the failure in diagnostics
- do not silently export dirty data
- do not silently proceed with stale or partial enrichment

### 3. Historic + live must become one continuous dataset
The backend must treat:
- historical backfill candles
- live streamed/aggregated candles
- derived timeframe candles

as part of one unified market history per symbol/timeframe.

No split-brain storage model.
No “historic over here, live over there” ambiguity.
No overwrite-based behavior that corrupts continuity.

### 4. This task is backend only
Do NOT:
- redesign UI pages
- retheme tabs
- clean up frontend labels
- patch UI wording except for minimal compatibility if an endpoint/shape must change and a compile fix is required

The proper UI rebuild comes later.

### 5. No strategy redesign in this task
Do NOT redesign:
- engine logic
- entry rules
- TP/SL philosophy
- allocator/coordinator behavior

This task is about making the data trustworthy and making AI/research plumbing valid.

### 6. No fake completeness
Do not claim a timeframe is supported unless it is truly:
- persisted
- or deterministically derivable on demand from stored lower timeframe data

Do not claim export is clean unless validation actually proves it.

### 7. Preserve existing platform stability
Do not break:
- Deriv streaming
- scheduler startup
- DB schema compatibility unless additive changes are required
- paper/demo/real separation
- V3 live engine runtime
- existing research storage unless it must be corrected for integrity

## Required implementation tasks

### 1. Stop destructive data pruning
Inspect the current code for any behavior that deletes valid historical candle data during:
- research runs
- backfill
- export preparation
- startup data sync
- maintenance jobs

If there is logic that effectively says “keep only 12 months” or otherwise discards valid older data from local storage, remove or replace it.

Requirements:
- valid available historical data must be retained
- retention rules, if any, must be explicit and non-destructive by default
- research/export/live logic must not depend on pruning to function

Return the exact files/modules changed for this.

---

### 2. Build a unified market-history model
Ensure the backend treats market data as one continuous stored dataset per symbol/timeframe.

Requirements:
- historical backfill data and live-derived data must coexist in the same truth model
- duplicate ingestion must be deduplicated safely
- overlapping inserts from historic/live sources must not corrupt continuity
- if additive schema metadata is helpful, it may be added minimally and explicitly

At minimum, verify and enforce clean uniqueness rules for:
- symbol
- timeframe
- timestamp

If current constraints are missing or insufficient, add the minimum safe fix.

Also audit tick storage:
- if ticks are vulnerable to duplicate insertion, add the appropriate uniqueness protection or deduplication mechanism

Fail loudly on conflicting writes that indicate corrupted logic.

---

### 3. Replace artificial history limits with “fetch all available”
Inspect the historical backfill logic and remove any artificial limit that restricts fetching to 12 months if the API can provide more.

Required new behavior:
- fetch all available history from the API endpoint
- do not hardcode “12 months” as the cap unless the API itself enforces it
- if the API is paginated, keep requesting until no more data is available
- persist all valid returned history

Do not blindly re-download all data every time.
The system must become gap-aware.

---

### 4. Add gap detection and targeted gap-fill
Create a backend integrity layer that can detect missing timestamps for stored candles.

At minimum support:
- 1m continuity checks
- correct expected interval checks for derived higher timeframes
- duplicate timestamp detection
- ascending-order validation

Required behavior:
- identify missing segments per symbol/timeframe
- request only missing ranges where practical
- fill gaps from the API for base data
- do not silently ignore missing history

This must work as reusable backend logic, not just inside export.

Make the integrity layer callable from:
- backfill/top-up workflows
- export preparation
- research preparation
- future diagnostics

---

### 5. Build deterministic timeframe enrichment
Create or formalize a deterministic candle enrichment pipeline.

Required supported timeframe set:
- 1m
- 5m
- 10m
- 20m
- 40m
- 1h
- 2h
- 4h
- 8h
- 1d
- 2d
- 4d

Requirements:
- 1m remains the base stored truth for candles if that is the current architecture
- higher timeframes must be deterministically derived from stored lower timeframe data
- define clearly which timeframe is the derivation base for each higher timeframe
- do not invent values
- preserve correct OHLCV aggregation semantics
- enrichment must be idempotent
- enrichment must not create duplicates
- enrichment must be rerunnable after gap fill

If 5m is already available directly from the API or stored separately, decide whether:
- to preserve it as a stored input
- or to rebuild it deterministically from 1m
Choose the safest/cleanest architecture and implement consistently.

At the end, the backend must be able to guarantee clean candle availability for the timeframe set above.

### 6. Add data top-up / reconciliation workflow
Create a reusable backend workflow that:
1. inspects what data exists
2. detects missing history/gaps
3. fetches missing base data from the API
4. repairs/inserts base candles
5. reruns timeframe enrichment
6. validates integrity again

This must become the foundation for future:
- research preparation
- export preparation
- future “top up data” UI action

Do not build the UI button yet.
Build the backend capability now.

### 7. Make research/export preparation integrity-aware
Update the research ZIP export flow so it never exports dirty or incomplete data silently.

New required export flow:
1. validate source data
2. if gaps/duplicates/ordering issues are found, run the repair/enrichment workflow where possible
3. validate again
4. only then export
5. if integrity still fails, fail loudly and report why

Requirements:
- export manifest must reflect the final exported truth
- validation output must accurately report duplicates, gaps, ordering, counts, and pass/fail
- export must preserve all raw candle field values correctly
- export must not drop candles silently
- export must not create misleading “clean” bundles from dirty data

If export is asked for more than one timeframe and some derived timeframe data is missing, it must first run enrichment.

### 8. Ensure exports are AI-safe and portable
The export output must be safe for downstream AI analysis anywhere, not just inside this app.

Requirements:
- clean JSON export remains primary
- data must be chronologically ordered
- timestamps must be normalized consistently
- symbol/timeframe/source metadata must be explicit
- no hidden reliance on app-specific assumptions for interpreting the file
- if multiple timeframes are exported, structure them clearly and deterministically
- if chunking is used, manifest and validation must still allow reconstruction of full continuity

The goal is: if the bundle is uploaded to ChatGPT or another analysis system, nothing important is lost or misrepresented.

### 10. Add AI research job foundation
Build a new backend research capability for AI to analyze stored market data directly.

This is not a UI task yet.
Build the backend job/function/service layer.

Required behavior:
- AI research runs on stored database data, not just backtest rows
- default analysis window = all available data up to 1 year rolling
- if more than 1 year exists, use the latest rolling 1 year by default
- if less than 1 year exists, use all available data
- AI research should be able to analyze by symbol and identify:
  - move frequency
  - move size
  - hold duration
  - likely useful timeframes
  - repeatable setups
  - expected firing frequency
  - drift/change in recent behavior vs older behavior
  - what appears promising / degrading

Do not build fancy prompt-chaining UI here.
Build the backend job foundation and structured output model so the later UI can call it cleanly.

The job should be asynchronous/background-capable if the current architecture supports that cleanly.
Do not promise immediate completion if the analysis is heavy.

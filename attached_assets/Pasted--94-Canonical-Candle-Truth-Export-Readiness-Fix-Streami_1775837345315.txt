#94 - Canonical Candle Truth, Export Readiness Fix, Streaming State Fix, and Research AI Alignment

## What & Why
Task #93 successfully exposed the backend through a temporary V3 Admin page, and that exposure revealed several important truths:

1. **Canonical candle/export readiness is still not correct**
   - After running Top Up + Enrichment, 1m export readiness can still say:
     “No 1m data found ... either timeframe is not enriched or no data exists”
   - This is unacceptable if raw 1m candles already exist in `Candles`.
   - Raw 1m export must work off the canonical candle store directly.

2. **Canonical candle storage truth is still not fully verified**
   - Historical candles are visible in `Candles`
   - Live ticks are visible in `Ticks`
   - It is still not proven clearly enough that:
     - live ticks are rolled into persisted canonical 1m candles
     - enrichment writes derived candles into the same canonical store
     - export reads from that same canonical store

3. **Top Up and Enrichment are too disconnected**
   - Running enrichment separately creates the risk of enriching dirty/incomplete data
   - The correct operational flow is:
     inspect data → repair/fill gaps → then enrich
   - The UI separation exposed the issue, but the real problem is backend workflow design

4. **Symbol state / streaming state is not represented correctly**
   - The current exposure implies 12 symbols are “streaming” or active in a confusing way
   - The real model should be:
     - 28 symbols are available / connected for research and data operations
     - only 4 active trading symbols are actually streamed live by default
     - each symbol should have a distinct backend state such as:
       - available
       - idle
       - streaming
       - disabled
   - Backend-level per-symbol streaming control is needed before final UI

5. **AI research is too generic and not aligned to system truth**
   - The current AI research output is useful only in a rough exploratory sense
   - It does not properly classify findings in terms of:
     - long-hold swing strategies
     - medium-hold strategies
     - spike / exhaustion / recovery logic
     - engine-compatible opportunities
     - expected trade frequency, win/loss, projected monthly return
   - It also does not clearly use the repo’s strategy source-of-truth documents as a governing frame

6. **Temporary UI feedback revealed backend gaps that must be fixed before final UI**
   - This task is NOT the final UI rebuild
   - This task fixes the backend issues exposed by #93 so the final UI can be built on truth

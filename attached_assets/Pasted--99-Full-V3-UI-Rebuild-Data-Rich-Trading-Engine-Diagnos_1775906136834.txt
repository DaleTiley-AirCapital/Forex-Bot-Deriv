#99 - Full V3 UI Rebuild: Data-Rich Trading, Engine Diagnostics, and Decision Transparency

## What & Why
The backend is now in a sufficiently trustworthy state to proceed with the full V3 UI rebuild.

Recent backend work established:
- canonical candle truth in the existing `candles` table
- top-up / reconcile can now recover real API candles correctly
- interpolated candle usage has been reduced drastically and is now truly last resort
- export range truth is functioning properly
- AI research is much closer to the required direction
- symbol streaming state is now clearer and more correct

We no longer want to keep patching the old V2-style UI.

This task is the **full V3 UI rebuild**:
- aligned to the backend as it now actually works
- data rich
- transparent
- diagnostic
- explanation-first
- suitable for live streaming + paper trading + ongoing strategy refinement

The user explicitly wants a UI that is numerically rich and operationally transparent.

That means:
- trades must show why they triggered
- signals must show why they did or did not pass
- engines must show expected move / confidence / rationale
- coordinator decisions must be visible
- allocator decisions must be visible
- hybrid trade lifecycle must be visible
- research and export must be practical and trustworthy
- the old V2 tabs / concepts should no longer dominate the app

This is a full user-facing rebuild, not another temporary admin patch.

---

## Non-negotiable rules

### 1. V3 only as the primary UI
The rebuilt UI must reflect the V3 architecture as the live truth.

Do NOT present the app as though the old V2 family-based system is still the main operating model.

If legacy pages/components remain temporarily, they must be:
- clearly marked legacy
- hidden from primary navigation
- not treated as current truth

### 2. Data-rich, not decorative
The UI must prioritize useful information:
- counts
- percentages
- rankings
- confidence
- trade rationale
- rejection reasons
- expected vs realized outcomes
- data quality / candle quality where relevant

Do NOT replace useful numbers with vague summaries.

### 3. Decision transparency is mandatory
Every meaningful decision surface must answer:
- what happened
- why it happened
- why it did not happen
- what the system expected
- what the current state is now

### 4. No fake or placeholder data
Only show fields that are actually backed by the backend.
If a field does not yet exist, either:
- expose the backend support in this task if it is small and necessary
- or omit it cleanly

Do NOT invent metrics.

### 5. Keep the current look and feel broadly consistent
Do not do an unrelated redesign.
Keep the same general visual language where practical, but rebuild the information architecture and components to match V3.

### 6. Minimal backend additions allowed only if needed for UI truth
This is primarily a UI task, but it is acceptable to add small backend support endpoints/fields if necessary to expose:
- engine reasoning
- rejection reasons
- coordinator decisions
- allocator outcomes
- hybrid stage state
- export range truth
- streaming state
- research results

Do not use this as an excuse for another backend redesign.

## 1. Overview page rebuild
The Overview page should become the operational dashboard.

It must show at minimum:
- current mode (paper/demo/real)
- streaming status summary
- active symbols
- engine activity summary
- open trades count
- pending candidates count
- last scan time
- last trade time
- capital allocation summary
- current paper/live PnL summary
- current data health summary for active symbols
- top current opportunities ranked by engine/coordinator output
- top current blockers / warnings

Include useful cards such as:
- Active Streaming Symbols
- Canonical Data Health
- Open Positions
- Candidate Signals
- Last Coordinator Decisions
- Research Freshness / latest AI run
- System Warnings

This page should answer:
- what is the system doing right now?
- what matters most right now?

## 2. Engine Decisions page (replaces old Signals worldview)
This is one of the most important pages.

This page must show decision flow per symbol and engine.

At minimum per decision row/card show:
- symbol
- engine name
- direction
- decision state:
  - triggered
  - candidate
  - rejected
  - suppressed
- confidence
- projected move %
- expected hold type:
  - short / medium / long / full-hold
- expected hold duration if available
- regime context
- coordinator outcome
- allocator outcome
- timestamp

Critically, also show:
- why it passed
- why it failed
- why it was suppressed
- threshold misses
- missing prerequisites
- conflict with stronger engine
- allocator rejection
- exposure constraint
- bad data / interpolated contamination if relevant

Add filters:
- symbol
- engine
- state
- mode
- date/time range

Support drill-down:
Clicking a decision should open a detailed panel showing:
- engine inputs/features
- threshold values
- coordinator comparison
- final verdict
- rejection reasons or trade handoff details

This page must make it easy to debug:
- why a “good looking” signal did not trade
- why a weak-looking one did

## 3. Trades page rebuild
Trades must be explanation-rich.

For each open/closed trade show:
- symbol
- engine origin
- coordinator decision origin
- entry timestamp
- entry price
- direction
- quantity / capital allocated
- expected move %
- TP
- SL
- trailing stop status
- current stage:
  - extraction
  - protection
  - runner
  - re-entry eligible / not eligible
- realized PnL
- unrealized PnL
- expected hold category
- actual hold time
- exit reason when closed

Also show trade rationale:
- why it was opened
- what the engine expected
- what invalidates it
- what stage transition occurred and why
- whether it is behaving as expected or underperforming expectation

Include views for:
- Open Trades
- Closed Trades
- Trade Lifecycle
- Stage Transitions
- Trade Attribution by Engine/Symbol

This page should let the user answer:
- what trades are live?
- why are they live?
- what are they trying to achieve?
- are they behaving correctly?

## 4. Research page rebuild
The Research page must become a real strategy/research workspace, not a vague status page.

It should support:

### A. Strategy Ranking View
Display ranked strategy candidates from the deterministic extractor / AI research pipeline, including:
- strategy name
- symbol
- family
- direction
- trades per month
- avg return %
- median return %
- win rate
- expected monthly return
- avg hold duration
- max drawdown if available
- capital efficiency
- current-engine fit:
  - aligned
  - partially aligned
  - research-only
  - rejected

### B. AI Research Results
Display the structured AI research report clearly:
- summary
- top strategies
- behavioral observations
- drift analysis
- why the AI agrees/disagrees with current engines
- whether opportunity structure appears stable or degrading

### C. Comparison Views
Support comparison of:
- symbol vs symbol
- current AI vs deterministic extractor
- current engine vs research opportunity
- recent period vs older period

### D. Export
Integrate trustworthy export UX here:
- symbol/timeframe selection
- real available date range
- selected-range row count
- real vs interpolated counts
- first/last date
- export action
- validation visibility

### E. Repair/Reconcile/Enrich controls
Keep operational data actions available here or in a Data/Admin area:
- top-up / reconcile
- enrich
- rerun AI research

Do not make the user go to a hidden debug page for core research operations.

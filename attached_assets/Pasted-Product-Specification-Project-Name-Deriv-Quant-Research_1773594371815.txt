Product Specification

Project Name

Deriv Quant Research & Execution Platform

Objective

Build a containerised research, backtesting, probability modelling, and live execution platform for Deriv synthetic indices, specifically Boom and Crash markets, with an emphasis on:
	•	low-frequency, high-conviction trade setups
	•	statistical modelling of historical tick behaviour
	•	regime detection
	•	coordinated multi-strategy execution
	•	aggressive but controlled capital deployment
	•	cashflow-focused operation rather than long-term passive compounding

The platform must be deployable on a Synology NAS via Docker/Container Manager and structured so it can later be adapted for real forex and crypto markets.

⸻

1. Business Context

Primary Use Case

The user wants to build an internal quantitative trading platform instead of buying third-party bots. The platform must:
	•	collect and store large volumes of Deriv historical and live tick data
	•	model market behaviour statistically
	•	backtest multiple strategies
	•	score trade opportunities probabilistically
	•	execute trades automatically when edge thresholds are met
	•	enforce strict portfolio and trade-level risk controls
	•	support periodic cash extraction

Trading Philosophy

This is not a high-frequency scalping platform.

The intended system characteristics are:
	•	20–30 trades per year per strategy, not hundreds per day
	•	trades held for hours or days
	•	each trade has predefined stop loss and take profit
	•	system may disable strategies for extended periods if market regime is unfavourable
	•	capital should be concentrated into the strongest opportunities
	•	when signal quality is high, the system should be able to deploy a large proportion of available capital
	•	objective is to maximise cash extraction, not theoretical Sharpe optimisation

3.3 Probability Model / Research Layer

This is the core differentiator.

The system must analyse historical Boom/Crash behaviour and produce probability scores for future trade opportunities.

Required Modelling Goals

The system must estimate probabilities such as:
	•	probability of rebound after consecutive adverse candles
	•	probability of trend continuation after pullback
	•	probability of breakout after compression
	•	probability of spike occurrence within next N ticks
	•	expected adverse excursion before favourable move
	•	expected reward-to-risk based on current market state

Required Feature Engineering

Include at least:
	•	candle body size
	•	wick ratios
	•	rolling volatility
	•	ATR
	•	Bollinger band width
	•	RSI
	•	EMA slope
	•	distance from mean
	•	consecutive candle counts
	•	run length since last spike
	•	ticks since last spike
	•	local trend state
	•	z-score of price vs rolling mean
	•	rolling skew / kurtosis if useful
	•	regime labels

Required Model Types

V1 should support at least:
	•	logistic regression baseline
	•	gradient boosting model
	•	configurable rule-based scoring fallback

Optional V2:
	•	hidden Markov regime model
	•	survival / hazard model for spike timing
	•	Bayesian transition model

Required Outputs

For each trade signal, the model should output:
	•	probability score
	•	signal type
	•	confidence band
	•	expected value estimate
	•	regime compatibility flag
	•	suggested trade direction
	•	suggested SL / TP parameters

3.4 Strategy Engine

The system must support multiple strategy modules operating under one portfolio engine.

Initial Strategy Families

Implement these four strategy families:

Strategy 1: Trend Pullback
Purpose:
	•	trade continuation after pullback in an established trend

Example logic:
	•	identify dominant trend
	•	wait for pullback to moving mean / support zone
	•	confirm momentum resumption
	•	enter continuation trade
	•	hold for hours/days

Strategy 2: Exhaustion Rebound
Purpose:
	•	trade mean reversion after an overstretched move

Example logic:
	•	2–4 consecutive candles against short-term mean
	•	price stretched from rolling average
	•	momentum weakening
	•	enter reversal / rebound setup

Strategy 3: Volatility Breakout
Purpose:
	•	trade expansion after compression

Example logic:
	•	Bollinger squeeze / ATR compression
	•	breakout from defined range
	•	confirm directional momentum
	•	ride move with fixed or trailing exit

Strategy 4: Spike Hazard / Spike Capture
Purpose:
	•	model elevated likelihood of a Boom/Crash spike event

Example logic:
	•	use tick features and time-since-last-spike
	•	score probability of spike within next N ticks
	•	enter only when hazard score exceeds threshold

Strategy Rules

Each strategy must:
	•	support enabling/disabling
	•	support symbol-specific tuning
	•	support configurable confidence threshold
	•	support hard SL/TP
	•	support max hold duration
	•	return a structured signal object, not place trades directly

⸻

3.5 Portfolio Signal Router

This is critical.

The system must not just let four bots trade independently and fight for capital.

Instead, strategies must submit signal candidates into a portfolio allocation engine which:
	•	ranks signal quality
	•	checks regime fit
	•	checks risk budget
	•	checks current open exposure
	•	decides which signal(s) are allowed
	•	sizes positions dynamically
	•	can concentrate capital into the best opportunities

Required Behaviour

If one strategy has a very strong signal:
	•	allow large capital deployment

If several weak signals exist:
	•	allow none or small allocation only

Allocation Modes

Support:
	•	conservative
	•	balanced
	•	aggressive

Example logic:
	•	weak score: deploy 10–20% of available capital
	•	medium score: deploy 30–50%
	•	strong score: deploy 60–90%

⸻

3.6 Trade Execution Engine

The execution engine must connect to Deriv and:
	•	authenticate via API token
	•	request market data
	•	submit trade proposal if required by Deriv API flow
	•	place trade
	•	store full order metadata
	•	monitor trade state
	•	manage SL/TP lifecycle
	•	close/exit where supported
	•	log execution events and errors

Requirements
	•	retries on temporary API failures
	•	idempotent order handling
	•	full audit log
	•	separate paper trading mode and live mode
	•	environment variable controls for live trading enablement

⸻

3.7 Risk Manager

The risk layer must be first-class, not bolted on.

Required Controls

Trade-level:
	•	max risk per trade
	•	max size per trade
	•	max hold time
	•	required SL on every trade
	•	required TP on every trade

Portfolio-level:
	•	max simultaneous trades
	•	max open risk
	•	max daily loss
	•	max weekly loss
	•	max monthly drawdown
	•	strategy cooldown after losing streak
	•	strategy disable on regime mismatch
	•	portfolio-wide kill switch

Cash Extraction Logic

Support:
	•	fixed withdrawal threshold
	•	account cap
	•	profit sweep rule

Example:
	•	when equity grows to threshold X, log withdrawal recommendation
	•	optionally trigger notification to manually withdraw

9. Strategy Logic Detail

Important Constraint

This platform must not fire constant tiny trades.

The expected operational profile is:
	•	few high-conviction trades
	•	larger capital deployment
	•	hours/days holding period
	•	hard stop and hard target
	•	only trade when score / regime justifies it

Signal Selection Rules

Every signal should pass:
	•	regime filter
	•	minimum model score
	•	expected value threshold
	•	minimum reward/risk threshold
	•	portfolio capital availability
	•	risk policy check

15. Acceptance Criteria

The project is accepted when:
	1.	It can be deployed on Synology via Docker Compose
	2.	It can pull and store Deriv historical and live tick data
	3.	It can derive candles and engineered features
	4.	It can run backtests across at least 4 strategy families
	5.	It can train and score a probability model
	6.	It can rank signals and allocate capital across strategies
	7.	It can paper trade end-to-end
	8.	It can run live with hard risk controls
	9.	It logs all trades, signals, and metrics
	10.	It exposes a dashboard and API for monitoring

16. Critical Design Principles for the Developers

The developers must understand the following non-negotiables:
	•	This is not a simple indicator bot
	•	This is a quant research and execution framework
	•	Strategies do not directly place trades
	•	Portfolio allocation is central
	•	Risk manager has override authority over all strategies
	•	Strategy frequency must stay low and high-conviction
	•	Capital utilisation should be dynamic and concentrated
	•	Regime detection is essential
	•	System must be designed to support future forex and crypto expansion

⸻

17. Questions the Developers Must Solve During Build

The development team must explicitly address:
	•	best way to label spike events from raw ticks
	•	best candle aggregation and timeframe mix
	•	whether SQLite is sufficient for V1 on Synology or PostgreSQL should be used from day one
	•	best model serialization and retraining process
	•	how live order lifecycle is handled on Deriv
	•	how to handle API interruptions and safe restart
	•	how to implement walk-forward testing cleanly
	•	how to structure config so strategies can be tuned without code edits

18. Summary Brief for the Agencies

One-paragraph version

Build a Dockerised Python application for Synology that ingests Deriv Boom/Crash tick data, generates candles/features, trains probability models, backtests multiple low-frequency swing-style strategies, ranks live trade opportunities by expected value and regime fit, allocates capital across strategies dynamically, and executes paper/live trades with strict portfolio risk controls, full logging, and an internal dashboard.
